# Database Documentation

## Overview

The Drivo admin dashboard interacts with **two data stores**:

1. **Firebase Firestore** — managed directly from the frontend for real-time notifications.
2. **Drivo Backend Database** — managed exclusively through the backend REST API. The frontend has no direct database access. Schema is inferred from API responses.

---

## Part A — Firebase Firestore

### Connection
- **Project:** `drivo-project-6f3fd`
- **SDK:** `firebase/firestore` v12
- **Access:** Direct from browser using hardcoded config in `src/lib/firebase.js`
- **Rules:** Not visible in this repository — must be configured in the Firebase Console

### Collection: `notifications`

This is the **only Firestore collection** used by the application.

#### Document Structure

| Field | Type | Description |
|---|---|---|
| `title` | `string` | Notification title displayed in the UI |
| `body` | `string` | Notification message body |
| `driverId` | `string` | Associated driver ID (empty string for system notifications) |
| `type` | `string` | `"driver"` or `"system"` |
| `read` | `boolean` | Whether the notification has been read |
| `createdAt` | `Timestamp` | Firestore server timestamp |

#### Example Document
```json
{
  "title": "سائق جديد انضم للمنصة",
  "body": "محمد السالم انضم كسائق جديد",
  "driverId": "abc123",
  "type": "driver",
  "read": false,
  "createdAt": "Timestamp(2026-06-24T10:30:00Z)"
}
```

#### Indexes Required
- `createdAt DESC` — used by `subscribeNotifications()` and `syncBackendNotifications()`
- `read == false` — used by `markAllAsRead()`

#### Operations Used
| Operation | Firestore Method | Called From |
|---|---|---|
| Add notification | `addDoc` | `saveNotification()`, `sendDriverNotification()`, `initNotificationsCollection()`, `syncBackendNotifications()` |
| Read all | `getDocs` | `markAllAsRead()`, deduplication in `syncBackendNotifications()` |
| Real-time subscribe | `onSnapshot` | `subscribeNotifications()` → `Layout.jsx`, `useUnreadCount.js` |
| Mark one as read | `updateDoc` | `markAsRead()` |
| Mark all as read | `updateDoc` (batch) | `markAllAsRead()` |

#### Deduplication Strategy
To prevent duplicates when syncing from the backend API, notifications are keyed by:
```
key = title + "|" + driverId
```
This is fragile — two different notifications with the same title and driver ID will be deduplicated into one.

## Data Retention & Privacy Notes
- Driver identity documents (photos) are stored server-side. Retention policy unknown.
- IBAN and banking information are stored in the `drivers` table. Access restricted to admin/accountant roles.
- No encryption-at-rest is applied at the frontend layer.
- Firebase Firestore security rules are not visible in this repository and must be reviewed separately.
